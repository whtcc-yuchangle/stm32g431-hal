# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

STM32G431RBTx (Cortex-M4, 128KB Flash / 32KB SRAM) embedded application built with STM32CubeMX 5.6.1 HAL and Keil MDK 5.35 (ARM Compiler 5). The hardware is a CT117E-M4 development board with a 240×320 TFT LCD, 4 buttons, 8 LEDs, AT24C02 EEPROM, and MCP40xx digital pot on bit-banged I2C.

## Build

Open `MDK-ARM\HAL.uvprojx` in Keil uVision 5.35+. The project outputs `MDK-ARM\HAL\HAL.hex` (Intel Hex, enabled). Compiler defines: `USE_HAL_DRIVER;STM32G431xx`. ARM Compiler 5 (not AC6).

System clock: HSE → PLL (÷3, ×20, ÷2) = **160 MHz**, Flash latency 3WS, voltage scale 1.

## Architecture

### Code generation boundaries (critical)

The project is STM32CubeMX-generated. **All user code lives between `USER CODE BEGIN` / `USER CODE END` markers.** Code outside these markers is regenerated by CubeMX when the `.ioc` file is modified. Never place custom logic outside USER CODE blocks.

### Page system (UI framework)

The application uses a simple linked-list page manager (`Src/page/page_manager.c`):

- `page_register(pt_page)` — prepends a page to the global list
- `page_get("NAME")` — finds a page by name string
- `page_init()` — registers all pages at boot
- `page->run()` — enters the page's event loop (blocks until back key)

Each page (`Src/page/*_page.c`) follows the same pattern:
1. Declare icon arrays, label arrays, and persistent state as `static`
2. Implement a `static void xxx_page_run(void)` — **infinite loop** reading keys and updating LCD
3. Declare a `static t_page g_xxx_page` with `.name` and `.run`
4. Provide `int xxx_page_register(void)` calling `page_register(&g_xxx_page)`

The active page is set once in `main()`: `page = page_get("MAIN");` then `page->run()` in the while(1) loop. Pages navigate to each other by calling `page_get("OTHER")->run()`.

### Key input

Defined in `Src/gpio.c`:

| GPIO | Key | Return value |
|------|-----|-------------|
| PB0  | KEY1 | 1 (next / up) |
| PB1  | KEY2 | 2 (prev / down) |
| PB2  | KEY3 | 3 (enter / confirm) |
| PA0  | KEY4 | 4 (back / return) |

`key_scan(&index)` does 50ms debounce and blocks until key release. Returns 0 if no key pressed.

### Peripherals

| Peripheral | Pins | Purpose |
|-----------|------|---------|
| USART1 | PA9(TX), PA10(RX) | printf output via `fputc` redirect at 9600 baud |
| ADC1 / ADC2 | — | Dual ADC sampling, `adc_get_value()` returns raw + voltage |
| DAC1 CH1/CH2 | — | 0–3.3V output, `dac1_set_value(ch, voltage)` |
| TIM16/TIM17 | — | PWM output (CH1), duty 10–90%, `TIMx->CCR1` direct write |
| TIM2/TIM3 | — | Input capture for frequency/duty measurement |
| Software I2C | PB6(SCL), PB7(SDA) | Bit-banged via `Src/i2c_hal.c` — not hardware I2C |
| LCD | PC8–PC15 (data), PB5/8/9/PA8/PD2 (control) | 240×320 TFT, 16-bit parallel interface, ILI9341-compatible |
| LEDs (×8) | PC8–PC15 | Active-low via 74HC573 latch (PD2) |

### Image assets

`res/*.bmp` → `convert_bmp24to16.cpp` → `Src/logo.c` (`const unsigned char` arrays as RGB565). The converter runs as a standalone C++ program: `g++ convert_bmp24to16.cpp -o convert && ./convert res/xxx.bmp`. The output C array replaces or appends to `Src/logo.c`.

### AI page (UART classification)

`Src/page/ai_page.c` receives classification results from a host PC over USART1 interrupt at 9600 baud. Protocol: ASCII string `"class:N"` (7 bytes, N = 0–9 digit).

| Class | Display |
|-------|---------|
| 0–2 | Fruit image (apple/banana/orange) + label |
| 3 | Label only ("grape", no image) |
| 9 | Label only ("unknown", no image) |

**Display window caveat:** `LCD_SetDisplayWindow` restricts GRAM writes to a region. After any operation that sets a restricted window (icon drawing or area clearing), the window must be restored to full screen (`LCD_SetDisplayWindow(239, 319, 240, 320)`) before drawing labels or text at other positions — otherwise text outside the window region will not appear.

### printf caveat

`fputc` in `Src/usart.c` calls `HAL_UART_Transmit` (blocking), so `printf` in interrupt/timer callbacks will cause hard faults or data corruption. Keep printf usage to main loop and page run functions only.
